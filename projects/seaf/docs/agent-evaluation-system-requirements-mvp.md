# 通用智能体评测系统需求规格说明书（MVP 版）

> 文档版本：V1.1  
> 日期：2026-04-19  
> 状态：待评审  
> 定位：最小可行版本，快速上线，可扩展  
> 更新说明：增加工作流智能体评测支持

---

## 一、背景与目标

### 1.1 背景

随着大语言模型（LLM）应用的快速普及，企业级 AI 智能体正在成为主流交互形态。然而，智能体的质量保障面临严峻挑战：

| 痛点 | 描述 |
|------|------|
| 评测效率低 | 依赖人工逐条打分，评测集规模受限，无法支撑高频迭代 |
| 质量回归风险 | 每次模型升级或提示词变更，无法量化效果变化，导致上线后质量滑坡 |
| 评测维度单一 | 仅关注最终回答质量，无法区分工具调用、知识库召回、流程执行等不同层次的能力 |

### 1.2 目标

**核心目标**：构建一套**通用、可扩展、自动化**的智能体评测系统，实现：

1. **自动打分** — 去掉 80%+ 人工打分工作量
2. **版本回归** — 量化每次变更的效果变化
3. **多类型覆盖** — 支持推理智能体 + 工作流智能体评测
4. **快速上线** — MVP 版本控制在 3-4 周内完成

### 1.3 MVP 范围定义

| 包含 | 不包含（后续版本） |
|------|-------------------|
| 推理 Agent 评测 | 编排 Agent 评测 |
| **工作流 Agent 评测**（新增） | 编排 Agent 评测 |
| LLM 自动打分（工具调用 + 准确性） | 更多评测维度扩展 |
| 基础报告（通过率 + 分项得分） | 慢节点定位图表、趋势图表 |
| 版本对比视图 | 定时任务、人工复核队列 |
| Grade 评级（A/B/C/D/F） | 评测集分层、版本管理 |
| 告警通知（分数下降 > 10%） | CI/CD 集成、发布前钩子 |

## 二、智能体分类体系

### 2.1 MVP 阶段支持的类型

| 类型 | 核心机制 | MVP 评测重点 |
|------|----------|-------------|
| **推理智能体** | 单 Agent + 多工具协作（知识库/Skill/MCP） | 工具调用准确性、答案准确性 |
| **工作流智能体** | DAG 节点串联（顺序/并行执行） | 流程完整性、节点性能、端到端耗时 |

> ⚠️ 编排智能体评测在后续版本支持，本文档暂不覆盖。

### 2.2 推理智能体评测维度

| 评测维度 | 权重 | 说明 | 评测方式 |
|----------|------|------|----------|
| 工具调用准确性 | 60% | 是否调用正确的工具，参数是否正确 | LLM 评委打分 |
| 答案准确性 | 40% | 最终回答是否正确解决了用户问题 | LLM 评委打分 |

### 2.3 工作流智能体评测维度

| 评测维度 | 权重 | 说明 | 评测方式 |
|----------|------|------|----------|
| 流程完整性 | 30% | DAG 是否跑完整，核心节点是否全部执行 | 系统日志判定 |
| 节点性能 | 30% | 各节点耗时是否在合理范围内 | 性能指标采集 |
| 端到端耗时 | 20% | 从启动到结果返回的总时长 | 性能指标采集 |
| 结果质量 | 20% | 最终输出是否正确解决了用户问题 | LLM 评委打分 |

> 💡 工作流评测采用"系统判定 + LLM 打分"混合模式，流程完整性和节点性能由系统自动判定，无需人工介入。

## 三、功能需求

### 3.1 评测集管理

| 功能 | 描述 | 优先级 |
|------|------|--------|
| **评测集 CRUD** | 按智能体类型管理评测集，支持创建/编辑/删除/查询 | P0 |
| **评测集查询** | 支持按类型、标签筛选评测集 | P0 |
| **评测数据项管理** | 支持创建/编辑/删除评测数据项 | P0 |

**评测数据项结构（MVP 精简版）：**

```json
{
  "id": "唯一标识",
  "name": "评测项名称",
  "agent_type": "reasoning | workflow",
  "query": "用户问题/触发条件",
  "expected_tools": ["期望调用的工具列表（可选，仅推理类型）"],
  "expected_answer_keywords": ["期望答案关键词（可选）"],
  "expected_nodes": ["期望执行的节点列表（可选，仅工作流类型）"],
  "max_node_latency_ms": 3000,
  "max_total_latency_ms": 15000,
  "tags": ["标签1"],
  "difficulty": "easy | medium | hard"
}
```

> ⚠️ MVP 阶段：评测数据项结构精简，不预设 evaluation_dimensions，由系统根据 agent_type 自动关联评测维度和评测方式。

### 3.2 评测任务执行

| 功能 | 描述 | 优先级 |
|------|------|--------|
| **手动触发评测** | 选择评测集 + 智能体，执行评测任务，实时显示进度 | P0 |
| **评测进度查看** | 显示当前评测任务进度、耗时、已完成/总数 | P0 |
| **评测任务历史** | 记录历史评测任务，支持查询和重新查看结果 | P0 |

**评测流程（推理智能体）：**

```
用户选择评测集和智能体
    ↓
系统逐条读取评测数据项
    ↓
调用推理智能体，获取响应 + 工具调用日志 + 耗时
    ↓
调用 LLM 评委打分（直接采纳，不做置信度判断）
    ↓
汇总生成评测报告
```

**评测流程（工作流智能体）：**

```
用户选择评测集和工作流智能体
    ↓
系统逐条读取评测数据项
    ↓
触发工作流执行，采集节点执行日志（含耗时、状态）
    ↓
系统自动判定流程完整性 + 节点性能
    ↓
调用 LLM 评委对最终结果打分（仅结果质量维度）
    ↓
汇总生成评测报告
```

### 3.3 LLM 自动打分（核心）

| 功能 | 描述 | 优先级 |
|------|------|--------|
| **工具调用评分** | LLM 评委评测工具调用的正确性和参数准确性（仅推理类型） | P0 |
| **答案准确性评分** | LLM 评委评测最终回答的正确性 | P0 |
| **结果质量评分** | LLM 评委评测工作流最终输出质量（仅工作流类型） | P0 |
| **加权总分计算** | 根据权重计算加权总分 | P0 |
| **评委模型配置** | 支持配置 LLM 作为评委（默认推荐 DeepSeek-V3） | P0 |

**LLM 评委 Prompt（推理智能体）：**

```
【角色】
你是一个专业的 AI 评测专家。请对以下 Agent 的回答进行公正、严格的评分。

【输入信息】
用户问题：{user_query}
Agent 回答：{agent_response}
期望调用的工具：{expected_tools}（如有）
工具调用记录：{tool_call_log}（格式：tool_name + arguments + result）

【评分维度】

1. 工具调用准确性（权重 60%）
   - 是否选择了正确的工具
   - 传入工具的参数是否正确完整
   - 0分：工具调用完全错误或遗漏关键工具
   - 50分：调用了部分正确工具
   - 100分：工具调用完全准确

2. 答案准确性（权重 40%）
   - 回答内容是否正确解决了用户问题
   - 0分：完全错误或误导用户
   - 50分：部分正确，有明显错误
   - 100分：完全正确

【输出格式】（严格 JSON）
{
  "scores": {
    "tool_usage": 85,
    "accuracy": 90
  },
  "weighted_score": 87,
  "key_findings": "核心发现说明",
  "main_issues": ["问题1", "问题2"]
}

【注意事项】
- 必须严格按 JSON 格式输出，不包含任何额外文本
- 分数为整数（0-100）
```

**LLM 评委 Prompt（工作流智能体）：**

```
【角色】
你是一个专业的 AI 评测专家。请对以下工作流智能体的输出结果进行公正、严格的评分。

【输入信息】
用户问题：{user_query}
工作流最终输出：{workflow_output}
工作流节点执行记录：{node_execution_log}
  示例格式：
  [节点A: 耗时1.2s, 状态:成功] →
  [节点B: 耗时3.5s, 状态:成功]

【评分维度】

结果质量（权重 100%）
   - 最终输出是否正确解决了用户问题
   - 0分：完全错误或无输出
   - 50分：部分正确，有明显缺陷
   - 100分：完全正确

【输出格式】（严格 JSON）
{
  "scores": {
    "result_quality": 85
  },
  "weighted_score": 85,
  "key_findings": "核心发现说明",
  "main_issues": ["问题1"]
}

【注意事项】
- 必须严格按 JSON 格式输出，不包含任何额外文本
- 分数为整数（0-100）
```

### 3.4 工作流性能自动判定

| 功能 | 描述 | 优先级 |
|------|------|--------|
| **流程完整性判定** | 系统自动判定 DAG 是否跑完整，核心节点是否全部执行 | P0 |
| **节点性能判定** | 系统自动判定各节点耗时是否在阈值内，超时标记为失败 | P0 |
| **端到端耗时判定** | 系统自动判定总耗时是否超过阈值 | P0 |
| **失败节点定位** | 自动定位耗时最长或失败的节点 | P1 |

**性能判定规则：**

| 维度 | 判定规则 | 满分 | 扣分规则 |
|------|----------|------|----------|
| 流程完整性 | 核心节点全部执行 = 100%；跳过 1 个节点扣 30 分 | 100 | 每跳过 1 个核心节点 -30 |
| 节点性能 | 所有节点耗时 < max_node_latency_ms = 100% | 100 | 每超时 1 个节点 -20 |
| 端到端耗时 | 总耗时 < max_total_latency_ms = 100% | 100 | 超时 -50 |

### 3.5 版本对比

### 3.6 告警通知

| 功能 | 描述 | 优先级 |
|------|------|--------|
| **回归检测** | 新版分数下降超过阈值（默认 10%）→ 标记为回归 | P0 |
| **告警通知** | 检测到回归时，通过配置的通知渠道（站内/邮件）发送通知 | P0 |
| **告警阈值配置** | 支持按智能体类型配置分数下降告警阈值 | P1 |

### 3.7 Grade 评级

| Grade | 分数区间 | 含义 | 行动 |
|--------|----------|------|------|
| **A** | 90-100 | 优秀 | 直接发布 |
| **B** | 80-89 | 良好 | 建议优化后发布 |
| **C** | 70-79 | 合格 | 建议修复低分项后发布 |
| **D** | 60-69 | 不合格 | 必须修复后重新评测 |
| **F** | < 60 | 严重不合格 | 阻断发布，需紧急修复 |

**Grade 计算规则（简化版）：**
- 按加权总分对应 Grade 区间
- 任一维度得分 < 60 → Grade 不高于 C

## 五、非功能需求

| 需求 | 说明 |
|------|------|
| **性能** | 推理 Agent：≤ 3min（按 20 题）；工作流 Agent：≤ 10min（按 15 题） |
| **稳定性** | LLM 评委调用失败自动重试 3 次，仍失败则标记为失败 |
| **扩展性** | 新增智能体类型只需配置评测维度和评测方式，不改核心代码 |
| **数据隔离** | 不同租户/团队的评测数据完全隔离 |

## 六、技术架构要点

| 模块 | 技术选型 | 说明 |
|------|----------|------|
| **评测执行引擎** | 内部服务 | 负责调度 Agent 执行、日志采集、结果汇总 |
| **LLM 评委服务** | DeepSeek-V3（推荐） | 独立的评委服务，支持切换模型 |
| **性能采集服务** | 工作流日志采集 | 采集节点级别的耗时、状态数据 |
| **数据存储** | 评测集、评测结果、日志 | 支持关系型或文档型数据库 |
| **通知服务** | 站内通知 + 邮件 | 可扩展其他渠道 |

## 七、里程碑计划

| 阶段 | 内容 | 预计工时 | 目标 |

## 九、风险与应对

| 风险 | 可能性 | 影响 | 应对 |
|------|--------|------|------|

## 十、验收标准

| 功能类型 | 验收条件 |
|----------|----------|
| 推理 Agent - 评测集管理 | 可创建/编辑/删除/查询评测集，支持增删评测数据项 |
| 推理 Agent - 手动触发评测 | 选择评测集后可在 3min 内完成 20 题评测，并展示进度 |
| 推理 Agent - LLM 自动打分 | 每题返回工具调用分 + 准确性分 + 加权总分，无人工干预 |
| 工作流 Agent - 评测集管理 | 可创建/编辑/删除/查询工作流类型评测集 |
| 工作流 Agent - 手动触发评测 | 选择评测集后可在 10min 内完成 15 题评测，并展示节点执行日志 |
| 工作流 Agent - 性能自动判定 | 流程完整性、节点性能、端到端耗时会自动判定并计入分数 |
| 工作流 Agent - LLM 打分 | 最终输出由 LLM 评委打分（仅结果质量维度） |
| 基础报告 | 显示通过率、分项得分、Grade 评级 |
| 版本对比 | 可选择两个历史版本，对比多维分数变化 |
| 告警通知 | 分数下降 > 10% 时触发通知 |
| Grade 评级 | 按规则自动计算 Grade，任一维度 < 60 则 Grade ≤ C |

## 十一、术语表

| 术语 | 定义 |
|------|------|
| **推理智能体** | 单 Agent + 多工具协作的智能体类型 |
| **工作流智能体** | DAG 节点串联的流程执行智能体 |
| **LLM 评委** | 使用 LLM 自动评估 Agent 回答质量的技术 |
| **工具调用** | Agent 调用外部工具（API/MCP/Skill）完成特定任务 |
| **流程完整性** | DAG 是否跑完整，核心节点是否全部执行 |
| **节点性能** | 各节点耗时是否在合理范围内 |
| **回归检测** | 检测新版相对于旧版是否有质量下降 |
| **Grade** | 综合评级，分为 A/B/C/D/F 五档 |
