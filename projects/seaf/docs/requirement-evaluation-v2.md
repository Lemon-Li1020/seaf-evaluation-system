## 一、背景与目标

### 1.1 现状问题

Seaf 平台目前已具备基础的智能体评测能力，但存在以下核心痛点：

| 痛点 | 影响 |
|------|------|
| 评测打分依赖人工 | 评测集规模受限，无法高频执行 |
| 无版本回归机制 | 每次发版无法量化效果变化，存在上线风险 |
| 评测维度单一 | 仅评测最终响应结果，无法区分工具调用、知识库召回、编排合理性等不同能力 |
| 评测粒度不足 | 无法按智能体类型（推理/蜂群/工作流）差异化评测 |

### 1.2 优化目标

**V2 核心目标**：实现智能体评测的**自动打分**和**版本回归**，去掉 80% 人工打分工作量。

## 二、评测体系设计

### 2.1 智能体分类

Seaf 智能体分为三种类型，每种类型评测重点不同：

| 类型 | 核心机制 | 核心评测点 |
|------|----------|-----------|
| **推理智能体** | 单 Agent + 多工具协作 | 工具调用准确性、知识库召回、意图识别、多轮对话 |
| **蜂群智能体** | 多 Agent 编排协作 | 编排合理性、调用顺序、结果聚合、死循环检测 |
| **工作流智能体** | DAG 节点串联 | 节点性能、流程完整性、熔断降级、并发效率 |

### 2.2 评测维度体系

每种类型覆盖以下评测维度：

| 维度 | 推理 | 蜂群 | 工作流 | 自动化程度 |
|------|------|------|--------|------------|
| 工具调用准确率 | ✅ | ✅ | ⚠️ | 高 |
| 意图识别 | ✅ | ⚠️ | ❌ | 高 |
| 知识库召回 | ✅ | ⚠️ | ⚠️ | 中 |
| 编排合理性 | ❌ | ✅ | ❌ | 中 |
| 节点响应时间 | ❌ | ⚠️ | ✅ | 高 |
| 流程完整性 | ❌ | ⚠️ | ✅ | 高 |
| 幻觉检测 | ✅ | ⚠️ | ❌ | 中（LLM评委） |
| 多轮上下文 | ✅ | ✅ | ❌ | 中 |

### 2.3 评测集设计

评测集按智能体类型分类管理，每条评测数据包含：

```json
{
  "id": "唯一标识",
  "name": "评测项名称",
  "agent_type": "推理 | 蜂群 | 工作流",
  "query": "用户问题/触发条件",
  "expected_answer": "期望答案（如有）",
  "expected_tools": "期望调用的工具列表",
  "evaluation_dimensions": [
    { "dimension": "维度名称", "weight": 0.4 }
  ],
  "performance_baseline": { "p95_latency": "10s" },
  "tags": ["标签1", "标签2"],
  "difficulty": "easy | medium | hard"
}
```

**评测集规模规划（V2 阶段）：**

| 类型 | V2 目标题数 | 说明 |
|------|------------|------|
| 推理智能体 | 30-50 题 | 覆盖主要工具类型和典型场景 |
| 蜂群智能体 | 15-20 题 | 覆盖编排、冲突、死循环等 |
| 工作流智能体 | 15-20 题 | 覆盖性能、容错、并行等 |

## 三、功能需求

### 3.1 评测集管理（增强）

| 功能 | 描述 | 优先级 |
|------|------|--------|
| 评测集 CRUD | 按类型管理评测集，支持批量导入/导出 | P0 |
| 评测集版本管理 | 评测集变更记录，支持回滚 | P1 |
| 评测集分层 | 回归集（发布前必跑）/ 扩展集（定时全量） | P2 |

### 3.2 评测任务执行

| 功能 | 描述 | 优先级 |
|------|------|--------|
| 手动触发评测 | 选择评测集 + 智能体，执行评测任务 | P0 |
| 自动触发评测 | 智能体发布前钩子，不合格阻断上线 | P1 |
| 定时评测 | 每日/每周自动跑全量评测集 | P2 |

### 3.3 LLM 自动打分（P0 核心）

**打分流程：**

```
Agent 返回响应
    ↓
LLM 评委（DeepSeek-V3）
    ↓
多维打分（准确性/工具调用/完整性/格式/安全）
    ↓
置信度判断
    ↓
置信度高 → 直接采纳
置信度低 → 推送人工复核队列
```

**置信度机制：**

| 置信度 | 阈值 | 处理方式 |
|--------|------|----------|
| 高置信 | > 90% | 直接采纳 |
| 中置信 | 70-90% | 采纳，标记关注 |
| 低置信 | < 70% | 推送人工复核 |

**评分稳定性**：同一评测项连续跑 3 次，标准差 > 15 → 强制人工复核

### 3.4 版本回归对比（P1）

| 功能 | 描述 | 优先级 |
|------|------|--------|
| 版本对比视图 | 新版 vs 旧版多维打分对比，差异高亮 | P0 |
| 回归检测告警 | 分数下降超过阈值（如 10%）→ 阻断发布 + 通知 | P0 |
| 趋势图表 | 各版本分数趋势折线图 | P1 |

### 3.5 结果展示

| 功能 | 描述 | 优先级 |
|------|------|--------|
| 评测报告总览 | 通过率、总分、分项得分、 grade | P0 |
| 分类型报告 | 按推理/蜂群/工作流生成独立报告 | P1 |
| 慢节点定位 | 工作流评测中耗时超阈值节点 TOP N | P1 |
| 人工复核队列 | 置信度低的评测结果，支持在线打分 | P2 |

## 四、非功能需求

| 需求 | 说明 |
|------|------|
| **性能** | 单次评测任务 < 5min（按 50 题计算），支持并发评测 |
| **稳定性** | LLM 评委调用成功率 > 99%，失败自动重试 |
| **扩展性** | 支持后续接入新的评测类型和评分模型 |
| **数据安全** | 评测结果数据隔离，不同团队不可互见 |

## 八、风险与依赖

| 风险 | 应对 |
|------|------|
| LLM 评委打分不稳定 | 多次采样取中位数，设置置信度兜底 |
| 评测集建设成本高 | 优先覆盖高频场景，逐步扩充 |
| 依赖 LLM API 稳定性 | 配置降级策略，人工兜底 |
